# SQL День 3: Финальный лайвкодинг и контроль качества данных

## Цель дня

- Довести SQL до состояния «спокойно решаю задачи на собесе под давление».
- Тренировать не только синтаксис, но и проверку корректности результата.
- Закрыть типовые задачи: n-й максимум, сверка totals, аналитическое мышление.

## План дня (6-8 часов)

1. `1.5 ч` — повтор базовых паттернов (SELECT/JOIN/GROUP).
2. `2.5 ч` — контрольные задачи в тренажёре.
3. `1.5 ч` — упражнения на объяснение хода мысли.
4. `1-2.5 ч` — мок-лайвкодинг под таймер.

## 1) Типовые паттерны финального дня

### 1.1 N-й максимум

```sql
SELECT order_id, total_amount
FROM orders
ORDER BY total_amount DESC
LIMIT 1 OFFSET 1;
```

### 1.2 Проверка консистентности totals

```sql
SELECT
  o.order_id,
  o.total_amount,
  SUM(oi.quantity * oi.unit_price) AS calc_total
FROM orders o
JOIN order_items oi ON o.order_id = oi.order_id
GROUP BY o.order_id, o.total_amount
HAVING o.total_amount != SUM(oi.quantity * oi.unit_price);
```

### 1.3 Топ-N

```sql
SELECT user_id, SUM(amount) AS total_paid
FROM orders
WHERE status = 'paid'
GROUP BY user_id
ORDER BY total_paid DESC
LIMIT 3;
```

## 2) Как проверять себя после написания запроса

1. Результат по смыслу совпадает с условием?
2. Количество строк ожидаемое?
3. Нет ли подозрительных дублей?
4. Проверены ли крайние случаи (пустые/NULL значения)?
5. Сортировка/лимит соответствуют задаче?

## 3) Лайвкодинг-алгоритм на собесе

1. Перескажи задачу интервьюеру.
2. Озвучь допущения (если условие не полное).
3. Напиши минимально рабочий запрос.
4. Проверяй и улучшай шагами.
5. Проговори итог и ограничения.

## 4) Антиошибки под стрессом

- Не писать сразу «монолитный» запрос на 20 строк.
- Не менять 5 частей запроса одновременно.
- Не молчать на 2-3 минуты.
- Не спорить с постановкой задачи — лучше уточнить допущения.

## 5) Частые факапы и как их не допустить

1. Перепутал `ASC`/`DESC`.
- Лечение: после запроса взглядом проверить первую строку.

2. Потерял строки из-за `INNER JOIN` вместо `LEFT JOIN`.
- Лечение: всегда проговаривать, нужна ли полная левая выборка.

3. Неверно посчитал агрегат из-за дублей.
- Лечение: смотреть кардинальность после JOIN.

4. Путаешь `WHERE` и `HAVING`.
- Лечение: запомнить этапы выполнения запроса.

## 6) Финальная подборка задач для репетиции

1. Найти пользователей без заказов.
2. Вывести сумму paid заказов по пользователям.
3. Найти второй по величине заказ.
4. Найти заказы, где total не совпадает с суммой items.
5. Вывести топ-3 пользователей по сумме оплат.

## 7) Как отвечать, если запрос дал ошибку

Сильная реакция:

- «Проверю синтаксис и изолирую проблему шагами. Сначала оставлю FROM/JOIN, затем добавлю фильтр и агрегаты. Так быстрее найду источник ошибки».

Слабая реакция:

- «Не знаю, почему не работает» и молчание.

## 8) Финальный pre-interview SQL чеклист

- Уверенно пишу SELECT/WHERE/ORDER/LIMIT.
- Уверенно пишу INNER/LEFT JOIN и anti-join.
- Уверенно пишу GROUP BY/HAVING/SUM/COUNT.
- Могу объяснить каждый запрос словами.
- Могу отладить запрос за 2-3 минуты.

## 9) Стратегия на день собеса

1. Перед интервью: 20-30 минут SQL-разминки.
2. Пройти 2 короткие задачи (одна базовая, одна join/aggregate).
3. Повторить фразы для проговаривания хода мысли.

Ключевая мысль:

На техничке оценивают не только «правильный ответ», но и инженерное рассуждение, аккуратность и способность спокойно дойти до решения.
