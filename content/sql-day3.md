## 1) Как обычно проходит SQL-блок на собесе

Чаще всего дают 1-2 задачи, например:

1. Простая выборка (`SELECT/FROM/WHERE/ORDER BY/LIMIT`).
2. JOIN + агрегат (`SUM/COUNT/GROUP BY/HAVING`).

Иногда добавляют:

- «второй максимум»;
- «найди несоответствия данных»;
- «найди пользователей без заказов».

Важно: оценивают не только ответ, но и ход мысли.

## 3) Алгоритм «Скелет → Уточнение → Проверка»

## 3.1 Скелет

```sql
SELECT ...
FROM ...
JOIN ... ON ...
WHERE ...
GROUP BY ...
HAVING ...
ORDER BY ...
LIMIT ...;
```

## 3.2 Уточнение

Заполняешь только нужные части (не всё сразу).

## 3.3 Проверка

Смотришь:

- типы данных;
- количество строк;
- граничные случаи (`edge cases` — крайние случаи);
- не потерялись ли записи.

## 4) Типовые финальные задачи с детальным разбором

## 4.1 Второй максимум (`second highest value`)

### Условие

Найти заказ со второй по величине суммой.

### Решение

```sql
SELECT order_id, total_amount
FROM orders
ORDER BY total_amount DESC
LIMIT 1 OFFSET 1;
```

### Проверка

- сортировка точно `DESC`;
- `OFFSET 1` действительно пропускает самый большой.

---

## 4.2 Проверка консистентности total (`data consistency check`)

### Условие

Найти заказы, где `orders.total_amount` не равен сумме позиций `order_items`.

### Решение

```sql
SELECT
  o.order_id,
  o.total_amount,
  SUM(oi.quantity * oi.unit_price) AS calc_total
FROM orders o
JOIN order_items oi ON o.order_id = oi.order_id
GROUP BY o.order_id, o.total_amount
HAVING o.total_amount != SUM(oi.quantity * oi.unit_price);
```

### Что это проверяет

Это классический QA-запрос на бизнес-целостность данных.

---

## 4.3 Пользователи без оплаченных заказов

```sql
SELECT u.user_id, u.username
FROM users u
LEFT JOIN orders o
  ON u.user_id = o.user_id
  AND o.status = 'paid'
WHERE o.order_id IS NULL;
```

### Важный нюанс

Фильтр `status = 'paid'` внутри `ON` нужен, чтобы корректно сохранить логику `LEFT JOIN`.

---

## 4.4 Топ-5 пользователей по обороту

```sql
SELECT user_id, SUM(amount) AS total_paid
FROM orders
WHERE status = 'paid'
GROUP BY user_id
ORDER BY total_paid DESC
LIMIT 5;
```

## 4.5 Дубликаты (`duplicates`) по email

```sql
SELECT email, COUNT(*) AS cnt
FROM users
GROUP BY email
HAVING COUNT(*) > 1;
```

Это популярная задача на качество данных.

## 5) Пагинация на собесе: что сказать грамотно

Если спрашивают про пагинацию (`pagination`):

> Для offset-based пагинации (`пагинация на LIMIT/OFFSET`) обязательно нужен стабильный `ORDER BY`. Иначе страница 2 сегодня и завтра может дать разный набор строк.

### Пример

```sql
SELECT order_id, created_at, amount
FROM orders
ORDER BY created_at DESC, order_id DESC
LIMIT 20 OFFSET 40;
```

## 6) Частые ловушки под давлением

## 6.1 WHERE и HAVING перепутаны

Если условие по агрегату (`SUM`, `COUNT`) — почти всегда нужен `HAVING`.

## 6.2 LEFT JOIN случайно превращён в INNER JOIN

Если поставить фильтр по правой таблице в `WHERE`, можно потерять строки без пары.

## 6.3 Плохо задан ORDER BY в топах

Просили «топ», а сортировка `ASC`.

## 6.4 Неправильный `NULL`-чек

Нужно `IS NULL`, не `= NULL`.

## 7) Как проверять запрос до того, как отдать ответ

Чек-лист самопроверки:

1. Явно ли указана таблица в `FROM`?
2. Верный ли ключ в `JOIN ON`?
3. Там, где нужна фильтрация строк, использован `WHERE`?
4. Там, где нужна фильтрация групп, использован `HAVING`?
5. Если есть `LIMIT/OFFSET`, есть ли стабильный `ORDER BY`?
6. Понимаю ли я каждую возвращаемую колонку?

## 8) Как объяснять решение интервьюеру (готовый скрипт)

> Я беру таблицы `users` и `orders`, соединяю их по `user_id`. Потом фильтрую нужные строки в `WHERE`, считаю метрику через `SUM/COUNT`, группирую по ключу и сортирую результат. В конце проверяю, что количество строк и бизнес-логика совпадают с условием задачи.

Этот скрипт создаёт ощущение зрелого подхода.

## 12) Финальный чек «готов / не готов»

Готов, если ты:

- без паузы объясняешь `FROM` и пагинацию;
- решаешь базовые SQL-задачи без внешней помощи;
- умеешь отладить запрос, если ошибся;
- можешь проговорить ход решения спокойно и структурно.

Главная мысль:

На техничке выигрывает не тот, кто «не ошибается никогда», а тот, кто **быстро и грамотно мыслит в данных**.
