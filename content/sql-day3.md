## 1) Формат SQL лайвкодинга на QA-собесе

Чаще всего спрашивают:

- простая выборка + фильтр;
- join + агрегат;
- задача на «второй максимум»;
- задача на проверку качества данных.

Обычно это не олимпиадный SQL, а практичные задачи.

## 2) Поведение кандидата, которое даёт плюс

1. Переформулировать задачу своими словами.
2. Проговорить допущения.
3. Писать запрос по этапам.
4. Проверять результат и объяснять его.
5. Спокойно исправлять ошибки.

## 3) Техники решения под таймер

### 3.1 Метод «Скелет -> Детали»

Сначала:

```sql
SELECT ...
FROM ...
JOIN ...
```

Потом добавляешь:

- WHERE;
- GROUP BY;
- HAVING;
- ORDER BY;
- LIMIT/OFFSET.

### 3.2 Метод «Проверка кардинальности»

После JOIN всегда задай себе вопрос:

- сколько строк должно быть?
- не появилось ли неожиданных дублей?

### 3.3 Метод «Локальная валидация»

- сравни 2-3 строки вручную;
- проверь крайние значения;
- при агрегатах перепроверь одну группу вручную.

## 4) Классические задачи финального дня

## 4.1 N-й максимум

### Вариант: второй максимум

```sql
SELECT order_id, total_amount
FROM orders
ORDER BY total_amount DESC
LIMIT 1 OFFSET 1;
```

Проверка себя:

- точно DESC?
- OFFSET верный для второго места?

## 4.2 Контроль целостности totals

```sql
SELECT
  o.order_id,
  o.total_amount,
  SUM(oi.quantity * oi.unit_price) AS calc_total
FROM orders o
JOIN order_items oi ON o.order_id = oi.order_id
GROUP BY o.order_id, o.total_amount
HAVING o.total_amount != SUM(oi.quantity * oi.unit_price);
```

Смысл:

- найти заказы, где сумма позиций не совпадает с total заказа.

## 4.3 Топ пользователей по оплатам

```sql
SELECT user_id, SUM(amount) AS total_paid
FROM orders
WHERE status = 'paid'
GROUP BY user_id
ORDER BY total_paid DESC
LIMIT 5;
```

## 4.4 Пользователи без оплаченных заказов

```sql
SELECT u.user_id, u.username
FROM users u
LEFT JOIN orders o
  ON u.user_id = o.user_id
  AND o.status = 'paid'
WHERE o.order_id IS NULL;
```

Важный нюанс:

- фильтр по статусу добавлен в `ON`, чтобы сохранить корректную логику LEFT JOIN.

## 5) Частые ловушки и как их обходить

### Ловушка 1: условие статуса в WHERE после LEFT JOIN

Это может превратить LEFT JOIN в INNER JOIN.

Лечение:

- условия на правую таблицу в некоторых случаях переноси в ON.

### Ловушка 2: COUNT(*) vs COUNT(column)

- `COUNT(*)` считает все строки.
- `COUNT(column)` не считает NULL в этом столбце.

### Ловушка 3: дубли из-за many-to-many

Лечение:

- проверка ключей и условий ON;
- иногда нужен DISTINCT или предварительная агрегация.

## 7) Пошаговая отладка ошибки

Если запрос упал:

1. Проверяешь синтаксис (запятая, алиасы, скобки).
2. Упрощаешь до минимального SELECT.
3. Возвращаешь части по одной.
4. Локализуешь проблемный блок.
5. Фиксишь и перепроверяешь результат.

Что говорить вслух:

- «Я сейчас изолирую проблемную часть запроса и верну её по шагам».
