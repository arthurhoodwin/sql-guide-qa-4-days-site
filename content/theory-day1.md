# День 1: Теория тестирования (Testing Theory — теория тестирования)

- Отработать устные формулировки терминов и практическое применение, а не только определения.
- Закрыть блоки: виды тестирования, уровни тестирования, техники тест-дизайна, приоритизация тестов по рискам.

## План дня (пример на 8-10 часов)

1. `2.5 ч` — теория тестирования и терминология.
2. `2.5 ч` — техники тест-дизайна с примерами.
3. `2 ч` — мини-практика: проектирование тестов по фиче.
4. `1-2 ч` — повторение, квиз, устные ответы вслух.

## 1) Что такое тестирование и зачем оно нужно

**Testing (тестирование)** — это деятельность по оценке качества продукта через проверку требований, поведения и рисков.

Что важно проговорить на собесе:

- тестирование не доказывает отсутствие дефектов;
- тестирование снижает риск, а не гарантирует идеал;
- QA работает не только с багами, но и с предотвращением проблем на ранних этапах.

### Ключевые цели тестирования

- найти дефекты **Defect/Bug (дефект/баг)** до пользователя;
- подтвердить соответствие требованиям **Requirements (требования)**;
- дать прозрачную оценку рисков релиза **Release Risk (релизный риск)**;
- поддержать принятие решений о выпуске.

## 2) Уровни тестирования (Test Levels — уровни тестирования)

1. **Unit Testing (модульное тестирование)**
- Проверка отдельной функции/класса/модуля.
- Обычно зона ответственности разработчиков.
- Быстрое, дешёвое в запуске.

2. **Integration Testing (интеграционное тестирование)**
- Проверка взаимодействия компонентов.
- Частые дефекты: несовпадение форматов, протоколы, тайминги.

3. **System Testing (системное тестирование)**
- Проверка системы целиком как black box.
- Охватывает end-to-end сценарии.

4. **Acceptance Testing (приемочное тестирование)**
- Проверка готовности по бизнес-критериям.
- Опора на **Acceptance Criteria (критерии приемки)**.

## 3) Виды тестирования (Test Types — виды тестирования)

### Functional Testing (функциональное тестирование)

Проверяем, что система делает то, что требуется:

- позитивные сценарии **Positive Cases (позитивные кейсы)**;
- негативные сценарии **Negative Cases (негативные кейсы)**;
- граничные сценарии **Edge Cases (пограничные кейсы)**.

### Non-Functional Testing (нефункциональное тестирование)

Проверяем, как система работает:

- **Performance Testing (тестирование производительности)**;
- **Load Testing (нагрузочное тестирование)**;
- **Stress Testing (стресс-тестирование)**;
- **Security Testing (тестирование безопасности)**;
- **Usability Testing (тестирование удобства)**;
- **Compatibility Testing (совместимость браузеров/устройств/ОС)**.

## 4) Smoke / Sanity / Retest / Regression

1. **Smoke Testing (дымовое тестирование)**
- Быстрая проверка критического пути: «сборка жива?». 

2. **Sanity Testing (санити-тестирование)**
- Узкая проверка конкретной изменённой зоны.

3. **Retest (повторная проверка фикса)**
- Проверка конкретного исправленного дефекта.

4. **Regression Testing (регрессионное тестирование)**
- Проверка, что новые изменения не сломали старый функционал.

Как отвечать на вопрос «Retest vs Regression»:

- Retest: «Проверяю, что конкретный дефект исправлен».
- Regression: «Проверяю, что рядом ничего не сломалось».

## 5) Severity vs Priority

- **Severity (серьезность)** — насколько баг влияет на систему/пользователя.
- **Priority (приоритет)** — насколько срочно чинить по бизнесу.

Сильный пример:

- Опечатка на главной перед PR-кампанией: низкая Severity, но высокая Priority.
- Падение редкой background-задачи без влияния на клиента: высокая Severity внутри системы, но Priority может быть ниже, если есть обходной путь.

## 6) Техники тест-дизайна (Test Design Techniques — техники проектирования тестов)

### 6.1 Equivalence Partitioning (разбиение на классы эквивалентности)

Пример: поле возраста `18..65`.

- валидный класс: `18..65`;
- невалидные: `<18`, `>65`.

Логика: тестируем представителей классов, а не каждое значение.

### 6.2 Boundary Value Analysis (BVA — анализ граничных значений)

Для диапазона `18..65` тестируем:

- `17, 18, 19`;
- `64, 65, 66`.

Частая формулировка на собесе: «баги живут на границах».

### 6.3 Decision Table Testing (таблица решений)

Применяется, когда результат зависит от комбинации условий.

Пример:

- Premium пользователь?
- Сумма заказа > 5000?
- Первый заказ?

По каждой комбинации фиксируем ожидаемый результат.

### 6.4 State Transition Testing (переходы состояний)

Подходит для сущностей со статусами: заказ, платёж, заявка.

Что проверять:

- допустимые переходы;
- запрет недопустимых переходов;
- поведение при повторном событии (идемпотентность).

### 6.5 Pairwise Testing (попарное тестирование)

Нужно, когда параметров много и полный перебор слишком дорог.

Идея: покрыть все пары значений параметров, уменьшив количество тестов.

### 6.6 Error Guessing (предугадывание ошибок)

Основано на опыте QA:

- пустые/нулевые значения;
- необычные форматы;
- очень длинные строки;
- повторные клики/повторные запросы;
- пограничные тайминги.

## 7) Риск-ориентированный подход (Risk-Based Testing — риск-бейзд)

Когда времени мало, на техничке ожидают именно это:

1. Определить критические бизнес-флоу.
2. Выделить зоны максимального ущерба при дефекте.
3. Приоритизировать тесты: high risk -> medium -> low.
4. Зафиксировать непокрытые риски и сообщить команде.

## 8) Мини-практика на день

Сценарий: «Форма регистрации».

Собери набор:

1. Классы эквивалентности для email/пароля/возраста.
2. Границы для длины пароля.
3. Таблица решений для промокода/гео/типа аккаунта.
4. Негативные проверки: пустые поля, пробелы, спецсимволы, уже существующий email.
5. Приоритизация: что проверишь первым при `30 мин` времени.

## 9) Как отвечать устно (шаблон)

Формат сильного ответа:

1. Короткое определение.
2. Где это применяешь в реальной работе.
3. Пример из фичи.
4. Риск/ограничение.

Пример:

- «Boundary Value Analysis (анализ граничных значений) применяю для полей с диапазоном. В регистрации для возраста 18..65 проверяю 17/18/19 и 64/65/66, потому что ошибки чаще на границах. Ограничение — техника не покрывает все логические ветки, поэтому дополняю таблицей решений».

## 11) Быстрый oral drill (устный прогон)

Проговори вслух:

1. «Что такое тестирование и его цель?»
2. «В чем разница между ретестом и регрессом?»
3. «Какие техники тест-дизайна ты используешь чаще всего?»
4. «Как режешь тестовое покрытие, когда времени мало?»

Если на любой вопрос отвечаешь неструктурно — повторить блок и проговорить снова.
