# День 2: Жизненный цикл, Agile, клиент-сервер и API

## Цель дня

- Научиться уверенно отвечать на блок «процессы + архитектура + API».
- Связывать теорию с практикой QA: что проверяем, где риски, как коммуницируем в команде.
- Подготовиться к логическим задачам по клиент-серверному взаимодействию.

## План дня (8-10 часов)

1. `2 ч` — SDLC/STLC и Agile-процессы.
2. `2 ч` — роли, сущности и артефакты в команде.
3. `2.5 ч` — клиент-сервер и API: концепт + ошибки + проверки.
4. `1.5-2.5 ч` — практические сценарии и устный прогон.

## 1) SDLC и STLC

### SDLC (Software Development Life Cycle — жизненный цикл разработки)

Типовые этапы:

1. Сбор и анализ требований.
2. Дизайн/архитектура.
3. Разработка.
4. Тестирование.
5. Релиз.
6. Поддержка.

Что важно на собесе:

- QA участвует не только на этапе тестирования;
- раннее вовлечение QA уменьшает стоимость дефектов;
- качество формируется на всех этапах, не только в конце.

### STLC (Software Testing Life Cycle — жизненный цикл тестирования)

1. Анализ требований.
2. Планирование тестирования.
3. Проектирование тестов.
4. Подготовка тестовых данных и окружения.
5. Выполнение тестов.
6. Завершение тестирования и отчётность.

## 2) Agile (Agile — гибкие методологии) и сущности

### Scrum (Скрам)

Роли:

- **Product Owner (владелец продукта)**;
- **Scrum Master (скрам-мастер)**;
- **Development Team (команда разработки, включая QA)**.

События (ceremonies):

- Sprint Planning (планирование спринта);
- Daily Scrum (ежедневный синк);
- Sprint Review (демо результата);
- Sprint Retrospective (ретро по процессу).

Артефакты:

- **Product Backlog (бэклог продукта)**;
- **Sprint Backlog (бэклог спринта)**;
- **Increment (инкремент)**.

### Kanban (Канбан)

- Ограничение WIP **(Work In Progress — работа в процессе)**.
- Визуализация потока задач.
- Фокус на пропускной способности и времени цикла.

## 3) Definition of Ready и Definition of Done

### DoR (Definition of Ready — критерии готовности задачи к разработке)

Хороший минимум:

- есть цель и бизнес-контекст;
- есть критерии приемки;
- понятны зависимости;
- понятны ограничения;
- можно оценить объём.

### DoD (Definition of Done — критерии завершённости)

Обычно включает:

- код-ревью завершено;
- тесты пройдены;
- дефекты нужного уровня закрыты;
- документация обновлена;
- фича готова к релизу по командному стандарту.

## 4) Клиент-серверная архитектура (Client-Server Architecture — клиент-сервер)

Базовая модель:

- **Client (клиент)**: отправляет запрос.
- **Server (сервер)**: обрабатывает запрос и возвращает ответ.
- **Database (база данных)**: источник/хранилище данных.

Типовой flow:

1. Пользователь нажал кнопку на фронте.
2. Клиент отправил HTTP-запрос.
3. Сервер проверил auth/валидацию/бизнес-логику.
4. Сервер обратился к БД/другим сервисам.
5. Ответ вернулся клиенту.
6. UI отобразил результат.

## 5) API (Application Programming Interface — программный интерфейс)

API для QA — это контракт между системами:

- структура запроса;
- структура ответа;
- обязательные/необязательные поля;
- коды ответа;
- правила ошибок.

### Что обязательно знать про HTTP

- `200 OK` — успешно.
- `201 Created` — ресурс создан.
- `400 Bad Request` — невалидный запрос.
- `401 Unauthorized` — не авторизован.
- `403 Forbidden` — авторизован, но нет прав.
- `404 Not Found` — ресурс не найден.
- `500 Internal Server Error` — ошибка сервера.

## 6) API проверки на собесе

### 6.1 Позитивные проверки

- Корректный запрос -> корректный код и тело ответа.
- Поля и типы соответствуют контракту.
- Бизнес-логика соблюдена.

### 6.2 Негативные проверки

- Невалидные данные.
- Пустые обязательные поля.
- Неверные типы.
- Нет токена/битый токен.
- Нет прав доступа.

### 6.3 Контрактные проверки

- Набор полей стабилен.
- Типы полей не «плавают».
- Формат ошибок единообразный.

### 6.4 Дополнительные важные вещи

- **Idempotency (идемпотентность)** для повторных запросов.
- Пагинация: limit/offset, дубли между страницами, стабильная сортировка.
- Timeouts и ретраи.

## 7) Логические задачи, которые могут дать

### Кейс: «Почему на фронте пустой список, а в БД данные есть?»

Логика ответа:

1. Проверить корректность endpoint/метода.
2. Проверить фильтры и query params.
3. Проверить auth/permissions.
4. Проверить тело ответа в API (вдруг там пусто).
5. Проверить маппинг поля на фронте.
6. Проверить кэш/версию клиента.

### Кейс: «Почему в UI дубли?»

- Дубли в БД?
- Ошибка JOIN на бэке?
- Повторный запрос клиента?
- Некорректный key в рендере списка?

## 8) Мини-практика дня

Сценарий: endpoint `/orders`.

Сделай таблицу проверок:

1. Успешное получение списка.
2. Невалидный токен -> `401`.
3. Нет прав -> `403`.
4. Неверный параметр фильтра -> `400`.
5. Пагинация (page1/page2) без дублей.
6. Стабильная сортировка по дате.
7. Пустой набор данных — корректный ответ без ошибок.

## 9) Как отвечать по процессу (сильная формулировка)

- «Я участвую в refinement заранее, чтобы выявлять неоднозначности требований и сокращать дефекты до разработки. В спринте веду риск-ориентированную приоритизацию тестов, а на выходе оцениваю готовность по DoD и остаточным рискам».

## 10) Чек-лист самопроверки

- Могу объяснить SDLC и STLC и где QA влияет на качество.
- Могу объяснить Scrum/Kanban без воды.
- Могу объяснить DoR и DoD на примерах.
- Могу объяснить клиент-серверный flow шаг за шагом.
- Могу назвать базовый API-набор проверок для любой ручки.
- Могу уверенно объяснить разницу `401` vs `403`.
