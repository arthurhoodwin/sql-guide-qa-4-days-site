## Блок A: Теория тестирования (Testing Theory — теория тестирования)

### 1. Что такое тестирование

**Testing (тестирование)** — это процесс проверки того, что продукт:
- делает то, что должен делать (соответствует требованиям);
- не делает того, чего делать не должен;
- стабильно работает в реальных условиях.

### 2. Основные цели тестирования

- Найти дефекты **Defect (дефект)** / **Bug (баг)** до пользователя.
- Снизить продуктовые и бизнес-риски.
- Подтвердить готовность к релизу **Release readiness (готовность к релизу)**.
- Дать прозрачную информацию стейкхолдерам **Stakeholders (заинтересованные стороны)**.

### 3. Уровни тестирования

1. **Unit testing (модульное тестирование)**
- Проверяется отдельная функция/класс/модуль.
- Обычно ответственность разработчика.
- Плюс: быстро и локально.

2. **Integration testing (интеграционное тестирование)**
- Проверяется взаимодействие модулей между собой.
- Частые баги: формат данных, контракт, тайминги.

3. **System testing (системное тестирование)**
- Проверка всей системы целиком как black box.
- Включает функциональность end-to-end.

4. **Acceptance testing (приемочное тестирование)**
- Проверка готовности с точки зрения бизнеса/заказчика.
- Часто завязано на **Acceptance Criteria (критерии приемки)**.

### 4. Виды тестирования по цели

1. **Functional testing (функциональное тестирование)**
- Проверяем, что фича реализована по требованиям.

2. **Non-functional testing (нефункциональное тестирование)**
- Проверяем «как» система работает.
- Основные подвиды:
  - **Performance testing (тестирование производительности)**
  - **Load testing (нагрузочное тестирование)**
  - **Stress testing (стресс-тестирование)**
  - **Security testing (тестирование безопасности)**
  - **Usability testing (тестирование удобства использования)**
  - **Compatibility testing (тестирование совместимости)**

### 5. Критичные термины для собеседования

1. **Smoke testing (дымовое тестирование)**
- Быстрая проверка критических сценариев после сборки/деплоя.
- Вопрос, на который отвечает: «Система вообще жива?»

2. **Sanity testing (санити-тестирование / проверка здравости изменений)**
- Узкая проверка конкретной области после точечного изменения.

3. **Retest (повторное тестирование исправления)**
- Проверка конкретного ранее найденного дефекта после фикса.

4. **Regression testing (регрессионное тестирование)**
- Проверка, что новые изменения не сломали существующий функционал.

**Ключевая разница Retest vs Regression:**
- Retest: «Починили ли конкретный баг?»
- Regression: «Не поломали ли что-то рядом?»

### 6. Severity vs Priority

- **Severity (серьезность дефекта)** — насколько сильно баг влияет на систему/пользователя.
- **Priority (приоритет исправления)** — насколько срочно баг надо чинить по бизнес-плану.

Пример:
- Опечатка на главной: низкая Severity, но иногда высокая Priority (например, PR-событие).

## Блок B: Техники тест-дизайна (Test Design Techniques — техники проектирования тестов)

### 1. Equivalence Partitioning (разбиение на классы эквивалентности)

Идея: входные данные делятся на группы, где поведение системы ожидаемо одинаково.

Пример: возраст `18..65`
- Валидный класс: `18..65`
- Невалидный класс 1: `<18`
- Невалидный класс 2: `>65`

### 2. Boundary Value Analysis (анализ граничных значений)

Идея: баги часто живут на границах.

Для диапазона `18..65` минимум набор:
- `17`, `18`, `19`
- `64`, `65`, `66`

### 3. Decision Table Testing (тестирование таблицей решений)

Используется, когда много комбинаций условий.

Пример условий для скидки:
- пользователь premium;
- сумма > 5000;
- первый заказ.

Комбинации условий дают явные ожидаемые результаты и уменьшают хаос в проверках.

### 4. State Transition Testing (тестирование переходов состояний)

Подходит, когда система имеет явные состояния.

Пример заказа:
- `created` → `paid` → `shipped` → `delivered`
- Негативный переход: `delivered` → `created` (должен быть запрещен)

### 5. Error Guessing (предугадывание ошибок)

Основано на опыте QA:
- пустые поля;
- спецсимволы;
- очень длинные строки;
- duplicate request (дублирующий запрос);
- нестабильная сеть.

## Блок C: SQL Полное повторение — База

### 1. Что такое SQL

**SQL (Structured Query Language — структурированный язык запросов)** — язык для работы с реляционными базами данных **Relational Database (реляционная база данных)**.

### 2. Базовые сущности

- **Table (таблица)** — набор строк с одинаковой структурой.
- **Column (столбец)** — поле определенного типа.
- **Row (строка / запись)** — одна сущность.
- **Primary Key (первичный ключ)** — уникальный идентификатор записи.
- **Foreign Key (внешний ключ)** — ссылка на запись в другой таблице.

### 3. Типы данных

- `INTEGER` (целое число)
- `REAL` (дробное число)
- `TEXT` (текст)
- `DATE` (дата)
- `BOOLEAN` (булево значение)

### 4. Базовые команды

1. **SELECT (выборка данных)**
```sql
SELECT product_name, price
FROM products;
```

2. **WHERE (фильтрация)**
```sql
SELECT *
FROM products
WHERE price > 10000;
```

3. **ORDER BY (сортировка)**
```sql
SELECT product_name, price
FROM products
ORDER BY price DESC;
```

4. **LIMIT (ограничение количества строк)**
```sql
SELECT *
FROM products
LIMIT 5;
```

5. **DISTINCT (уникальные значения)**
```sql
SELECT DISTINCT category
FROM products;
```

### 5. Агрегатные функции

- **COUNT (количество)**
- **SUM (сумма)**
- **AVG (среднее)**
- **MIN (минимум)**
- **MAX (максимум)**

```sql
SELECT COUNT(*) AS total_products,
       AVG(price) AS avg_price
FROM products;
```

### 6. Типовые ошибки новичка

- Пропуск `FROM`.
- Путаница между `=` и `IN`.
- Неправильный порядок `WHERE` и `ORDER BY`.
- Сравнение с `NULL` через `=` вместо `IS NULL`.
