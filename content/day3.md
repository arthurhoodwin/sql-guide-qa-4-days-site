## Блок A: Тестовая документация — подробно

### 1. Test Case (тест-кейс)

Обычно включает:
- ID
- Название
- Preconditions (предусловия)
- Steps (шаги)
- Test Data (тестовые данные)
- Expected Result (ожидаемый результат)
- Actual Result (фактический результат)
- Status (статус прохождения)

Когда нужен:
- сложная логика;
- высокие риски;
- необходимость аудита покрытия.

### 2. Checklist (чек-лист)

Короткий список проверок без детальной пошаговости.

Когда удобен:
- быстрые smoke/sanity прогоны;
- повторяющиеся проверки;
- поддержка и регресс по шаблону.

### 3. Test Plan (тест-план)

Содержит:
- scope (область тестирования)
- out of scope (что не тестируем)
- подход к тестированию
- риски
- ресурсы
- критерии входа/выхода

### 4. Bug Report (баг-репорт)

Сильный баг-репорт:
- Title (заголовок)
- Environment (окружение)
- Preconditions (предусловия)
- Steps to Reproduce (шаги воспроизведения)
- Actual Result (фактический результат)
- Expected Result (ожидаемый результат)
- Severity/Priority
- Attachments (вложения: скрин, видео, лог)

### 5. RTM (Requirements Traceability Matrix — матрица трассировки требований)

Назначение:
- связать requirement (требование) с test case (тест-кейсом);
- увидеть пробелы покрытия;
- повысить управляемость качества перед релизом.

## Блок B: SQL Полный повтор (под лайвкодинг)

### 1. Базовый каркас написания запроса

1. Что нужно вывести? (`SELECT`)
2. Из какой таблицы? (`FROM`)
3. Какие фильтры? (`WHERE`)
4. Нужна ли группировка? (`GROUP BY`)
5. Нужна ли фильтрация агрегатов? (`HAVING`)
6. Нужна ли сортировка? (`ORDER BY`)
7. Нужно ли ограничить результат? (`LIMIT`)

### 2. Частые типы SQL задач на собеседовании

1. Фильтрация по условию
2. N-й максимум / минимум
3. JOIN между таблицами
4. Пользователи без заказов
5. Поиск дубликатов
6. Проверка расхождения totals

### 3. Шаблоны для повторения

#### Дубликаты
```sql
SELECT email, COUNT(*) AS cnt
FROM users
GROUP BY email
HAVING COUNT(*) > 1;
```

#### N-й максимум
```sql
SELECT *
FROM orders
ORDER BY total_amount DESC
LIMIT 1 OFFSET 1; -- 2-й максимум
```

#### Расхождение суммы заказа и позиций
```sql
SELECT o.order_id,
       o.total_amount,
       SUM(oi.quantity * oi.unit_price) AS calc_total
FROM orders o
LEFT JOIN order_items oi ON o.order_id = oi.order_id
GROUP BY o.order_id
HAVING ABS(o.total_amount - SUM(oi.quantity * oi.unit_price)) > 0.01;
```

### 4. Анти-ошибки

- `NULL` сравниваем через `IS NULL` / `IS NOT NULL`.
- Не путать `WHERE` и `HAVING`.
- Проверять сортировку по `ASC`/`DESC`.
- При JOIN всегда проговаривать, почему выбран именно этот тип JOIN.
