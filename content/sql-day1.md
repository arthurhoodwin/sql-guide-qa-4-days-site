# SQL День 1: Полный фундамент (с нуля) для QA-собеседования

## Главная цель дня

- Научиться уверенно читать и писать базовые SQL-запросы без ступора.
- Понимать не только «как написать», но и «почему результат именно такой».
- Дойти до состояния, где ты спокойно объясняешь любой базовый запрос вслух интервьюеру.

## Результат к концу дня

Ты должен уметь без подсказки:

1. Писать `SELECT/FROM/WHERE/ORDER BY/LIMIT`.
2. Работать с `NULL`, `DISTINCT`, `COUNT`, простыми агрегатами.
3. Проверять себя на типовые ошибки (фильтр, сортировка, границы, пустые значения).
4. Проговаривать решение структурно за 30-60 секунд.

## 0) Мини-словарь SQL терминов

- **Row (строка)**: одна запись таблицы.
- **Column (колонка/столбец)**: один атрибут сущности.
- **Schema (схема)**: структура таблиц и связей.
- **Query (запрос)**: инструкция к базе данных.
- **Result Set (результирующий набор)**: строки, вернувшиеся после SELECT.
- **Predicate (предикат)**: логическое условие в WHERE/HAVING.

## 1) Логический порядок выполнения SELECT (очень важно)

Ментальная модель:

1. `FROM` — откуда берём данные.
2. `WHERE` — какие строки оставляем.
3. `GROUP BY` — как группируем.
4. `HAVING` — какие группы оставляем.
5. `SELECT` — какие поля выводим.
6. `ORDER BY` — как сортируем.
7. `LIMIT/OFFSET` — сколько строк возвращаем.

Почему это важно на собесе:

- помогает не путать WHERE и HAVING;
- помогает отлаживать запрос по шагам;
- показывает зрелое понимание SQL, а не «магии».

## 2) Базовый каркас запроса

```sql
SELECT column_1, column_2
FROM table_name
WHERE condition
ORDER BY column_1 DESC
LIMIT 10;
```

Как объяснять:

- сначала выбрал таблицу;
- потом отфильтровал строки;
- потом отсортировал;
- потом ограничил число строк.

## 3) WHERE глубоко: операторы и паттерны

### 3.1 Сравнения

```sql
WHERE price > 10000
WHERE rating >= 4
WHERE status = 'paid'
WHERE city <> 'Москва'
```

### 3.2 Диапазоны

```sql
WHERE price BETWEEN 1000 AND 5000
```

### 3.3 Множества

```sql
WHERE city IN ('Москва', 'Казань', 'Самара')
```

### 3.4 Поиск подстроки

```sql
WHERE product_name LIKE '%phone%'
```

### 3.5 Комбинации условий

```sql
WHERE status = 'paid' AND amount > 3000
WHERE city = 'Москва' OR city = 'Казань'
WHERE NOT (status = 'cancelled')
```

## 4) Работа с NULL (один из самых частых фейлов)

Запомни:

- `NULL` не сравнивается через `=` или `<>`.
- Используются только `IS NULL` / `IS NOT NULL`.

Правильно:

```sql
WHERE deleted_at IS NULL
```

Неправильно:

```sql
WHERE deleted_at = NULL
```

## 5) ORDER BY и LIMIT

### 5.1 Сортировка

```sql
ORDER BY price ASC   -- от меньшего к большему
ORDER BY price DESC  -- от большего к меньшему
```

### 5.2 Топ-N

```sql
SELECT product_name, price
FROM products
ORDER BY price DESC
LIMIT 3;
```

### 5.3 Пагинация (базовый вариант)

```sql
LIMIT 10 OFFSET 20
```

Интервью-заметка:

- всегда думай о стабильной сортировке перед LIMIT/OFFSET.

## 6) DISTINCT, COUNT, SUM, AVG

### 6.1 DISTINCT

```sql
SELECT DISTINCT category
FROM products;
```

### 6.2 COUNT

```sql
SELECT COUNT(*) AS total_rows
FROM products;
```

### 6.3 SUM / AVG

```sql
SELECT SUM(price) AS total_price, AVG(price) AS avg_price
FROM products;
```

Практический смысл для QA:

- быстро сверить, что данные загружены/не пропали;
- проверить отчёты и виджеты на фронте;
- делать sanity-check после миграций.

## 7) Типовые собесные задачи День 1 (и как их решать)

### Задача 1

«Покажи товары дороже 10000, отсортируй по убыванию цены».

```sql
SELECT product_name, price
FROM products
WHERE price > 10000
ORDER BY price DESC;
```

### Задача 2

«Покажи 2 самых дорогих товара».

```sql
SELECT product_name, price
FROM products
ORDER BY price DESC
LIMIT 2;
```

### Задача 3

«Сколько уникальных категорий есть в таблице?»

```sql
SELECT COUNT(DISTINCT category) AS categories_cnt
FROM products;
```

### Задача 4

«Покажи строки, где категория не заполнена».

```sql
SELECT *
FROM products
WHERE category IS NULL;
```

## 8) Частые ошибки новичка

1. `= NULL` вместо `IS NULL`.
2. Перепутал `ASC`/`DESC`.
3. Не учел фильтр по статусу (например, paid/cancelled).
4. Указал неправильную колонку в ORDER BY.
5. Написал `SELECT *`, где спрашивали конкретные поля.

## 9) Как отлаживать запрос, если «не то»

Алгоритм:

1. Убери WHERE и посмотри базовый набор.
2. Добавляй условия по одному.
3. Проверяй число строк после каждого шага.
4. Смотри несколько первых строк глазами.
5. Убедись, что сортировка/лимит соответствуют задаче.

## 10) SQL для QA: практическая роль

Зачем SQL нужен QA на техничке:

- валидация данных после действий в UI/API;
- проверка интеграций между сервисами;
- подтверждение источника проблемы (UI vs backend vs data);
- быстрая аналитика по дефекту.

## 11) Мини-чеклист «готов/не готов»

- Пишу базовые SELECT без подсказки.
- Не путаю NULL-логику.
- Стабильно решаю 4-5 задач уровня «фильтр + сортировка + лимит».
- Могу объяснить запрос по шагам вслух.

## 12) Режим тренировки под собес

1. Ставь таймер на 6-8 минут на задачу.
2. Перед написанием — 20 секунд проговори план.
3. После решения — 20 секунд объясни результат.
4. Если ошибка — отладка строго пошагово, без паники.

Ключевая мысль:

Базовый SQL блок почти всегда можно пройти, если ты даёшь понятную логику и аккуратно проверяешь себя.
