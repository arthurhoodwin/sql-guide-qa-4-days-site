## Как Пользоваться Этой Страницей

Этот раздел сделан как твой личный «боевой конспект».  
Режим работы:

1. Открываешь нужный блок (опыт, сильные стороны, кейсы, вопросы).
2. Сначала учишь **короткую версию** (15-30 секунд).
3. Потом подключаешь **полную версию** (1-3 минуты с деталями).
4. Перед собесом прогоняешь только чек-листы и готовые формулировки.

### Компания / Проект / Роль

- Компания: Redberry
- Проект: EatFit (доставка здорового питания)
- Период: ноябрь 2024 — апрель 2025 (6 месяцев)
- Роль: QA Trainee / помощник тестировщика

- Процесс: Scrum, спринты по 2 недели

### Зона Ответственности По Функционалу

- Личный кабинет: регистрация, логин, восстановление пароля, профиль, история заказов.
- Корзина: добавление/удаление, изменение количества, пересчет цены, промокоды, сохранение состояния.
- Checkout: адрес, время, оплата, комментарий, подтверждение заказа.
- Панель курьера: принятие заказа, отметка о доставке, маршрут на карте.

### API

- REST API бэкенда на Django.
- Методы: GET, POST, PUT, DELETE.
- Проверка кодов ответа, структуры тела, заголовков, авторизации.
- Негативные сценарии: невалидные данные, отсутствие токена, недоступные роли.

### SQL

- Запросы: SELECT, WHERE, JOIN, GROUP BY, COUNT, SUM.
- Проверки консистентности: корректность записи заказа, user_id, статус, сумма.

### Адаптив / Кросс-девайс

- Desktop: 1920x1080, 1366x768.
- Tablet: iPad.
- Mobile: iPhone SE, Pixel 5.
- Сверка с Figma по отступам, видимости элементов, поведению форм.

### Интеграция Карт

- Проверка загрузки карты.
- Проверка корректного построения маршрута.
- Проверка очистки маршрута при смене заказа.

### Фильтрация И Поиск

- Фильтры по категории, цене, калориям, тегам.
- Поиск по названию.
- Комбинации фильтров с применением pairwise (попарного тестирования).

### Тестовая Документация

- Smoke-чек-листы 7-10 критических сценариев.
- Баг-репорты по структуре: title, severity/priority, environment, steps, actual/expected, attachments.

### Ролевая Модель

- Роли: клиент, курьер, менеджер.
- Проверка, что пользователь видит только разрешенные зоны.
- Валидация, что запрещенные действия реально запрещены на API и UI.

## 4) Ключевые Кейсы И Баги (STAR Формат)

## Кейc 1: Промокод Считался Неверно Перед Релизом

### S (Situation)
За час до релиза в smoke обнаружил некорректное списание суммы при промокоде.

### T (Task)
Понять воспроизводимость, зафиксировать дефект и не допустить релиз с финансовой ошибкой.

### A (Action)
Собрал шаги воспроизведения, приложил видео, проверил связку UI + API + БД и отдал баг с высоким приоритетом.

### R (Result)
Релиз сдвинули на полдня, баг исправили до выката.  
Я добавил постоянную проверку промокодов в smoke-чек-лист.

## Кейc 2: Маршруты На Яндекс Картах Накладывались

### S
При переключении между заказами старый маршрут не очищался.

### T
Подтвердить баг и исключить ввод пользователя в заблуждение.

### A
Стабильно воспроизвел кейс, приложил запись, уточнил expected поведение, проверил повторно после фикса.

### R
Дефект исправлен, сценарий добавлен в регрессионный пул.

## Кейc 3: API Давал 200 Вместо 401

### S
Неавторизованный запрос на оформление заказа возвращал успех.

### T
Выявить нарушение безопасности/контракта API.

### A
Проверил через Postman без токена, задокументировал расхождение контракта.

### R
Исправили статус-код и контроль авторизации.

## Кейc 4: Адаптив На iPhone SE

### S
Кнопка «Заказать» выходила за экран.

### T
Устранить критичную проблему доступности ключевого действия.

### A
Сделал скрин/видео, указал девайс, viewport, шаги и влияние на checkout.

### R
Верстку поправили, критичный сценарий стал проходимым на мобильном.

## Кейc 5: Фильтр + Поиск Ломали Логику

### S
Чекбокс фильтра визуально активен, но фактически фильтр сброшен.

### T
Поймать рассинхрон состояния UI и фактического запроса.

### A
Прогнал комбинации pairwise, локализовал последовательность воспроизведения.

### R
Исправлен state-sync, кейс внесен в регресс.

## 5) Релизный Процесс (Как Рассказывать На Интервью)

### Короткая Версия

Релиз был раз в 2 недели. За день до релиза — freeze ветки release, я прогонял smoke 20-30 минут. Если критичных нет — релиз. После выката — быстрый post-release smoke на проде. Раз в месяц — полный регресс.

### Полная Версия

1. За 1 день до релиза фиксировали scope: только багфиксы.
2. Я запускал smoke по чек-листу критических user flow.
3. При красном smoke эскалировал проблему PM и dev.
4. После деплоя делал sanity/smoke на проде.
5. Раз в месяц делали расширенный регресс.
6. При необходимости релиз сдвигали, если риск для пользователей высокий.

## 6) Команда И Взаимодействие

### Как Объяснять Роль QA В Команде

QA не «финальный контролер», а участник цикла разработки:

- помогает уточнять требования до реализации;
- снижает риск дефектов до релиза;
- даёт прозрачную картину качества;
- помогает команде принимать решения по готовности.

### Что Ты Улучшил В Процессе

- Preview-ссылки на feature-ветки через Vercel.
- Идея отдельного стенда с моковыми ролями/статусами.
- Обновление smoke-чек-листа после критичного кейса.

## 7) Твои Сильные Стороны (С Доказательствами)

### 1. Внимательность

Не ограничиваюсь «работает/не работает», проверяю мелкие рассинхроны UI, тексты ошибок, граничные состояния.

### 2. Системное Мышление

Тестирую связками: UI → API → БД → права доступа, а не только один слой.

### 3. Упорство

Если баг плавающий, довожу до воспроизводимости и закономерности.

### 4. Коммуникация

Формулирую баг так, чтобы разработчик мог быстро повторить и исправить.

### 5. Обучаемость

Быстро вошел в продукт и начал находить баги в критичных сценариях.

## 9) Что Люблю И Что Не Люблю В QA

## Что Люблю

Нравится искать неочевидные баги и помогать команде выпускать качество без стыда перед пользователем.

## Что Не Люблю

Плавающие дефекты и релизные импровизации вечером в пятницу, но именно такие ситуации лучше всего прокачивают QA-дисциплину.

## 12) Готовые Ответы На Частые Вопросы

## Почему QA, А Не Разработка

Мне ближе ответственность за качество: находить уязвимые места до релиза, а не чинить последствия после. Понимание архитектуры от разработки у меня осталось, и это усиливает мои проверки.

## Почему Хочешь В VK

Хочу работать с продуктом, где много пользовательских сценариев, интеграций и ролей. Мне интересны сложные кейсы доступа, API и качества данных в высоконагруженной среде.

## Расскажи О Конфликте

Когда были разногласия по приоритету фикса, я приносил факты: влияние на пользователя, воспроизводимость, риски для релиза. Обычно это снимало эмоции и переводило обсуждение в плоскость данных.

## 14) Красные Флаги, Которых Лучше Избегать В Речи

- «Я просто кликал и находил баги случайно».
- «Я не очень понимаю API/БД».
- «Я не люблю писать документацию».
- «Я обычно жду, пока мне скажут, что тестировать».

### Чем Заменить

- «Я строю проверки от рисков и критичных user-flow».
- «Проверяю связку интерфейса, API и данных».
- «Стараюсь документировать так, чтобы команда экономила время на воспроизведении».
- «При неясных требованиях сначала уточняю критерии приемки и приоритеты».

## 15) Микро-Чеклист Перед Собесом (На Последние 24 Часа)

1. Повтори 60-секундный рассказ о себе.
2. Проговори 3 STAR-кейса с цифрами и результатом.
3. Освежи 401/403, smoke vs regression, severity vs priority.
4. Реши 5 SQL задач подряд (SELECT + JOIN + GROUP BY).
5. Подготовь 3 вопроса интервьюеру.
6. Проверь тон: уверенно, спокойно, без «воды».
